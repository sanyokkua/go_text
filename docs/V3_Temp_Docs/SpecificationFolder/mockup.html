GoText · design system & final screens

The whole UI, every widget, both themes

A single reference for the redesign: design tokens, the full widget gallery in light and dark, every dropdown/popup open, horizontal & vertical layouts, changed-word diff highlighting, and the regrouped settings. Each interactive widget is mapped to the Radix primitive that backs it, so very little stays custom. Stacks show added inferences, never estimated time.

Foundations

Design tokens

One token set, two themes. Every widget reads CSS variables, so light/dark is a single class on the root — no duplicated component styling. The teal/purple identity from today's app is preserved and modernized. The active theme is a user setting — Auto (follow OS, live), Light, or Dark — set in Settings · Appearance (tab 7).

◆ Settings — what changed in v3

1
Providers: kind + env-var keys

5 kinds (Ollama, LM Studio, Llama.cpp, OpenAI-compatible, Azure). API keys come from an environment variable — the app never stores secrets.

2
Logging is richer

Task logging (prompts), diagnostic app logging to a rotating file (level + size/backups/age/compress), and History (enable + max entries + clear) — independent toggles.

3
Azure handled as one kind

Deployment-in-path URL, Api-Key auth, optional api-version, live discovery — no separate client.

4
About & data vs About · Info

This section is paths + factory reset. The user-facing guide + Prompt Inspector live in the separate About · Info window (tab 9).

5
Appearance: theme mode

Auto (follow OS, live), Light, or Dark — set in Settings · Appearance, persisted (ui.theme), applied via the single root token class.

Implementation

What Radix gives you — and the little that stays custom

Radix Primitives cover all the behaviour-heavy, accessibility-sensitive widgets (focus, keyboard, ARIA, positioning). For searchable lists, pair Radix with cmdk. The rest is plain styled markup — no widget framework. This is how dropping MUI eliminates custom logic without losing the look.

◆ Why this is the right footprint

1
Accessibility for free where it's hard

Focus traps, keyboard nav, ARIA roles and collision-aware positioning are exactly the parts you don't want to hand-roll — Radix owns them.

2
Unstyled = your design, not Material

Radix ships no visuals, so the tokens on Tab 1 fully control the look. No override-fighting like MUI.

3
cmdk closes the one gap

Radix has no combobox/command primitive; cmdk (same ecosystem) handles the model search and ⌘K palette.

4
The custom set is tiny & static

Chips, cards, the diff highlighter and stack bar are presentational — no behaviour to maintain.

Help & transparency

About · Info — the in-app guide, the action/stack catalog, and the Prompt Inspector

Opened from the ℹ button next to settings. A plain-language manual of how the app works and what every setting does, the full list of built-in actions and your saved stacks, and — new — a Prompt Inspector that shows the exact prompt(s) sent to the LLM: system prompt, user prompt, parameters, and (for stacks) the per-inference composition. Reuses the real planner/composer, so what you see is what gets sent.

◆ What this screen adds

1
Plain-language guide

How the app works and what every setting changes; how to add/edit/remove a provider (keys come from env vars, never stored).

Full action & stack catalog

Every built-in action by category, plus your saved stacks with their steps and how they're applied (merge → inferences).
