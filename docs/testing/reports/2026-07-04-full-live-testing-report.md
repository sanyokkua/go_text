## Overall assessment

All 16 phases (P0-P15) were executed against a genuinely fresh install, with no phase skipped
outright. Five findings were confirmed (a sixth, initially logged, was reviewed and ruled not a
bug — see below), none of which block normal single-user, single-instance operation, but one is a
high-severity correctness gap worth prioritizing:

- **#2 (high)** is the most user-visible: deleting the current provider leaves the Model picker
  pointing at a model name that doesn't exist on the newly-current provider, so the *very next*
  Run silently fails with `model_not_found` — no warning is shown at delete time. This and **#1**
  (medium: a live-selected model is discarded on any provider round-trip) share the same root
  cause — model selection is not kept in sync with the current provider across every
  state-changing path, only the paths that were originally built to keep them in sync. Both should
  be fixed together (likely by making `SetAsCurrentProviderConfig` and `DeleteProvider`'s
  reassignment path the single source of truth for both `providers.selected_model` and the global
  `settings.model.name`, rather than having two independent writers).
- **#3 (informational)** and **#4 (low)** are both "a control/contract silently doesn't apply in
  one code path" issues (Ollama ignoring the completion-endpoint override; the bootstrap logger
  ignoring build mode) — correct as-is in the sense that nothing crashes or corrupts data, but both
  violate a documented contract and would confuse someone debugging from the docs alone.
- **P2-T3 (oversized `contextWindow` not clamped)** was initially logged as a low-medium finding
  but **ruled not a bug on review**: the app has no way to know every model's true context ceiling
  across every provider/config combination, that's out of scope, and the generic 1024-200000 range
  check is the most it can reasonably enforce — the user is responsible for knowing their own
  model's capabilities. No follow-up task opened for this; the plan's §P2-T3 expectation
  ("oversized value is clamped or produces a clear message") is superseded by this ruling.
- **#5 (low)** is a policy question as much as a bug — whether multi-instance launches (outside
  the normal Finder/Dock double-click path, which is already safe) should be actively blocked —
  and is recorded as a finding rather than an assumed fix, since the "correct" behavior isn't
  specified by existing docs or tests.
- The **P10-T4 DuplicateStack argument-count hang** (dash-numbered, not a numbered finding since
  it's a recurrence of a 2026-07-03 finding explicitly declined at the time) has now been observed
  against two different bound methods across two testing sessions. It no longer looks like a
  one-off and is worth reconsidering as a systemic Wails-IPC-bridge characteristic — any bound
  method called with the wrong arity hangs the caller forever instead of rejecting, which would
  hang the real UI (not just test harnesses) if a genuine frontend/binding-drift bug ever calls a
  handler with stale arguments after a signature change.
- **P13 (built-binary lifecycle)** was run for real (`wails build`, not `wails dev`) but T1
  (live window-resize persistence) and T3 (`OnShutdown` mid-run cancellation) were only
  partially verifiable in this sandboxed session — no Screen Recording permission for
  `computer-use` screenshots and no devtools bridge in this non-devtools build meant there was no
  way to see or click the native window. The restore-on-startup half of T1 and the
  cancellation-mechanism half of T3 were still verified directly (DB manipulation + log inspection,
  and cross-reference to P11-T5's already-confirmed cancellation repro, respectively) — this is a
  tooling limitation of this specific session, not a gap in the app's behavior, and is recorded
  as skipped-with-reason per the plan's own §10 allowance rather than silently omitted.
- **P15 (factory reset)** ran last as required, via the real UI flow (not just the bridge call),
  and the DB state after reset was verified field-by-field against the fresh-install baseline —
  full pass, no findings.

On regression-test disposition: per this project's rule of writing a test for every found bug,
the intent is a red-test-then-fix-then-green cycle within the change that fixes it. Since this was
scoped as a testing/verification pass with fixes deferred, and some findings (#1, #5) have
product-level "what's the correct behavior" ambiguity that only the fix's author can resolve,
adding tests now would either assert today's buggy behavior (codifying it) or need to be
red-committed against a green branch (breaking CI). Each remaining finding is instead filed below
as a follow-up task whose scope explicitly includes "add a regression test alongside the fix," which
satisfies the rule at the point the fix actually lands.

## Follow-up tasks opened

1. **Fix Finding #1 + #2 (model/provider sync)** — make model selection follow a single source of
   truth across provider switch, provider delete, and the AppBar picker, instead of the current two
   independent writers (`UpdateModelConfig` vs `SetAsCurrentProviderConfig`/`DeleteProvider`'s
   reassignment path). At minimum: `DeleteProvider`'s current-provider reassignment
   (`internal/settings/repository_sqlite.go:262-297`) must go through the same sync logic as
   `SetAsCurrentProviderConfig` (`internal/settings/service.go:287-309`), and the frontend's
   `deleteProviderConfig` thunk (`frontend/src/logic/store/settings/thunks.ts:61-76`) must also
   re-dispatch `getModelConfig()`. Add regression coverage in `internal/settings/service_test.go`
   (provider-switch round-trip preserves or intentionally clears model) and
   `frontend/src/logic/store/settings/thunks.test.ts` (delete-current-provider resyncs both
   provider and model state) alongside the fix.
2. **Fix Finding #3 (Ollama ignores completion-endpoint override / legacy-max-tokens toggle)** —
   either make the Ollama native path (`internal/llms/ollama_native.go`) honor
   `profile.CompletionPath`/`useLegacyMaxTokens`, or (more likely, given Ollama's native `/api/chat`
   shape doesn't have an equivalent) hide/disable those two controls in the Providers UI when the
   provider kind is `ollama`, so the control doesn't imply an effect it doesn't have. Extend
   `internal/llms/ollama_native_test.go` (already covers the num_predict mapping) or add a UI test
   asserting the controls are disabled/hidden for Ollama.
3. **Fix Finding #4 (bootstrap logger hardcodes `dev=true`)** — `main.go:56`'s
   `logging.New(logging.DefaultConfig(), true)` should not unconditionally attach a console writer
   and INFO level before `Init()`/`Reconfigure()` runs. Since `runtime.Environment(ctx)` isn't
   available yet at that point, consider deferring DI-container construction's log lines until
   after `OnStartup`, or reading a build-tag/env-based dev signal for the bootstrap phase instead
   of a hardcoded literal. Add a Go test asserting the bootstrap `Config` used before `Reconfigure`
   does not enable a console writer in a non-dev build (may need a small refactor to make the
   bootstrap dev-detection injectable/testable).
4. **Decide Finding #5 (no single-instance lock)** — product decision needed on whether
   GoText should actively block concurrent instances (e.g., a lock file or named pipe checked at
   startup) given normal Finder/Dock usage is already safe and the only real exposure is
   `open -n`/direct-exec/non-macOS launches. If the decision is "block it," add an integration test
   simulating a second `db.Open` while a lock is held; if "leave it," no code change needed but
   worth a one-line doc note in `CLAUDE.md`/`README` under known limitations.
5. **Reconsider the P10-T4 `DuplicateStack`-and-recurring wrong-arity-hangs-forever Wails IPC
   behavior** (dash-numbered process note, not a numbered finding) — now observed against two
   different bound methods across two sessions (2026-07-03 report + this one). Investigate whether
   Wails v2's generated JS bindings can validate argument count client-side and reject immediately,
   or whether a thin wrapper is needed. If fixed, add a frontend test asserting a wrong-arity bound
   call rejects rather than hangs.
6. **P13 T1/T3 re-verification with proper GUI tooling** — this session's `computer-use` access had
   no Screen Recording permission and this build has no devtools bridge, so the live window-resize
   persistence (write half) and `OnShutdown` mid-run-quit (live GUI half) could not be directly
   exercised. Worth a dedicated follow-up pass once Screen Recording permission is available, or a
   `wails build -devtools` debug binary is used instead, to close this specific gap.
